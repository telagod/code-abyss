# Coff0xc · 合规、架构与威胁建模

## 触发

安全架构、STRIDE、威胁建模、等保、PCI-DSS、GDPR、ISO27001、SOC2、CIS、NIST、数据分类、脱敏、DLP、隐私、基线、控制矩阵、安全评审、风险模型、控制项、审计证据、上线前评审。

## 边界

- 合规映射不是法律意见；需要法务/合规负责人确认。
- 不编造审计证据、控制状态或认证结论。
- 生产控制变更前确认 owner、影响范围和回滚。

## 能力矩阵

| 能力域 | 覆盖 | 要点 |
| --- | --- | --- |
| 架构评审 | 信任边界、数据流、身份、网络、日志、恢复 | 图和证据 |
| 威胁建模 | STRIDE、PASTA、attack tree、abuse case | 风险到控制 |
| 合规映射 | ISO27001、SOC2、PCI-DSS、GDPR、NIST、CIS | 控制矩阵 |
| 数据安全 | 分类分级、加密、脱敏、DLP、保留、删除 | 数据生命周期 |
| 隐私 | 最小化、目的限制、主体权利、跨境 | 需合规确认 |
| 安全基线 | OS、cloud、K8s、DB、CI/CD、endpoint | 可检测 |
| 成熟度 | 过程、工具、度量、owner、改进路线 | 分阶段落地 |

## 工作流

1. 输入收集：架构图、数据流、资产、身份模型、合规目标、控制现状。
2. 边界建模：系统边界、信任边界、第三方、数据分类。
3. 威胁分析：威胁、影响、现有控制、缺口、优先级。
4. 控制映射：把风险映射到框架控制项和证据需求。
5. 落地计划：owner、SLA、验证方式、监控指标、例外管理。
6. 复查输出：风险登记、控制矩阵、证据清单、残余风险。

## 验证清单

- 数据流和信任边界可追溯。
- 每个风险有控制、owner 和验证方法。
- 每个合规控制有证据或缺口状态。
- 隐私和法律结论标注需专业确认。
- 例外和残余风险有审批路径。

## 反模式

- 用通用合规清单替代实际架构评审。
- 把“计划实施”写成“已实施”。
- 没有证据就声称合规。
- 忽略数据生命周期和第三方边界。
